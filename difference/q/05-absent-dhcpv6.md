## 1. RFC 8415 Original Words

### Section 1 — Introduction

RFC 8415 (which obsoletes RFC 3315) defines DHCPv6:

> "**DHCPv6** can be used for **stateful address autoconfiguration**. A DHCP server assigns addresses to clients and maintains a record of the assigned addresses. **DHCPv6 can be used in conjunction with** the stateless address autoconfiguration defined in [RFC 4862]."
>
> — RFC 8415, Section 1

### Section 1.1 — Stateful Address Autoconfiguration

> "**Stateful address autoconfiguration** involves a server (DHCP server) that maintains a database of address assignments and can respond to requests from clients for addresses. **A host uses stateful address autoconfiguration when it requires the assignment of addresses through DHCPv6.**"
>
> — RFC 8415, Section 1.1

### Section 5.2 — Interaction with Router Advertisements

RFC 8415 also specifies how DHCPv6 interacts with IPv6 Neighbor Discovery:

> "**When a host receives a Router Advertisement with the M flag (Managed address configuration) set, the host SHOULD use DHCPv6 to obtain addresses.** When the O flag (Other configuration) is set, the host SHOULD use DHCPv6 to obtain other configuration information (e.g., DNS servers, SIP servers)."
>
> — RFC 4861, Section 4.2 (cross-referenced by RFC 8415)

### RFC 4862 — Section 1 — Introduction

RFC 4862 (IPv6 Stateless Address Autoconfiguration) acknowledges the complementary role of DHCPv6:

> "**Both stateful and stateless address autoconfiguration can be used.** In stateful autoconfiguration, the addresses and other configuration information are obtained from a server (e.g., DHCPv6). In stateless autoconfiguration, the addresses are generated by the host based on prefix information advertised by routers."
>
> — RFC 4862, Section 1

And:

> "**When the ManagedFlag is set, the host should use DHCPv6** to obtain addresses, in addition to any addresses generated via stateless autoconfiguration."
>
> — RFC 4862, Section 5.5.3

## 2. What F-Stack Does

### Configuration Interface

F-Stack's IPv6 configuration is entirely **static** and **manual**. The `config.ini` file provides:

```ini
[port0]
addr=192.168.1.2
netmask=255.255.255.0
broadcast=192.168.1.255
gateway=192.168.1.1

# IPv6 configuration (all manual)
addr6=fd00::2
prefix_len=64
gateway6=fd00::1
```

There is:
- **No `dhcpv6=yes` option**
- **No `dhcp6_client` configuration**
- **No DHCPv6-related parameters** (e.g., DUID, IAID, rapid commit, DNS server request)

### SLAAC-Only Autoconfiguration

F-Stack supports only **stateless address autoconfiguration** (SLAAC) via the FreeBSD sysctl:

```c
net.inet6.ip6.accept_rtadv = 2   /* Accept Router Advertisements */
```

When enabled, F-Stack will:
1. Listen for Router Advertisements
2. Extract Prefix Information options
3. Autoconfigure addresses using the advertised prefixes
4. Configure default routes from the RA

However, this is **limited to SLAAC only** — it does not handle:
- The **M (Managed) flag** in RAs → no DHCPv6 Solicit message is sent
- The **O (Other) flag** in RAs → no DHCPv6 Information-Request is sent
- **Stateful address assignment** → no DHCPv6-based address configuration
- **DNS server discovery via DHCPv6** → only RDNSS (RFC 6106) via RA is supported
- **SIP server, NTP server, or other DHCPv6 options**

### Source Code Evidence

A code search of the F-Stack repository confirms **zero occurrences** of:

- `dhcp6`
- `dhcpv6`
- `DHCPV6`
- `SOLICIT` (in DHCP context)
- `IA_NA` (Identity Association for Non-temporary Addresses)
- `IA_TA` (Identity Association for Temporary Addresses)
- `DUID` (DHCP Unique Identifier)

No GitHub issue or pull request in the F-Stack repository addresses DHCPv6 support.

### Developer Documentation

F-Stack developer documentation on Tencent Cloud (published 2019-08-16) explicitly states:

> *"F-Stack 不支持 DHCPv6，主要是因为 FreeBSD 本身的 DHCPv6 支持不够好，需要额外的第三方 dhcpclient 工具支持，所以 IPv6 地址信息需要手动配置。"*
>
> Translation: *"F-Stack does not support DHCPv6, mainly because FreeBSD itself does not have good enough DHCPv6 support, and requires additional third-party dhcpclient tools for support, so IPv6 address information must be manually configured."*
>
> — F-Stack developer documentation, Tencent Cloud (2019-08-16)

## 3. Impact Analysis

### Environments Requiring DHCPv6

DHCPv6 is **required** (SLAAC is insufficient) in the following environments:

| Environment | Why DHCPv6 Is Needed | F-Stack Impact |
|-------------|---------------------|----------------|
| **Enterprise networks** | Centralized address management, DNS assignment | Cannot auto-configure; requires manual setup |
| **Cloud providers (non-AWS)** | Some cloud providers use DHCPv6 for IPv6 assignment | Cannot obtain IPv6 addresses automatically |
| **ISP networks** | Prefix delegation (PD) for home routers | Cannot request prefix delegations |
| **Networks with M-flag-only** | Networks that only set the M flag (not A flag) in RAs | Cannot auto-configure at all (SLAAC disabled) |
| **Networks requiring DNS via DHCPv6** | No RDNSS support in RA, DNS only via DHCPv6 | Cannot discover DNS servers |
| **Compliance requirements** | Certain compliance frameworks require DHCPv6 | Non-compliant |

### What Works Without DHCPv6

| Feature | Works Without DHCPv6? | Mechanism |
|---------|----------------------|-----------|
| Link-local address autoconfiguration | ✅ Yes | Always-on (fe80::/10) |
| Global address via SLAAC | ✅ Yes | RA Prefix Information options |
| Default route via RA | ✅ Yes | RA Router Lifetime field |
| DNS via RDNSS | ✅ Yes | RA RDNSS option (RFC 6106) |
| DNS via DNSSL | ✅ Yes | RA DNSSL option (RFC 6106) |
| DHCPv6 stateful address | ❌ No | Not implemented |
| DHCPv6 prefix delegation | ❌ No | Not implemented |
| DHCPv6 other config (DNS, NTP, SIP) | ❌ No | Not implemented |
| DHCPv6 rapid commit | ❌ No | Not implemented |
| DHCPv6 reconfiguration | ❌ No | Not implemented |

## 4. Why F-Stack Doesn't Have DHCPv6

The root cause is architectural:

1. **FreeBSD lacks a built-in DHCPv6 client**: Unlike Linux (which has `systemd-networkd`, `NetworkManager`, `dhclient`, `dhcpcd`), FreeBSD's base system does not include a DHCPv6 client. The `dhclient` in FreeBSD's base only supports DHCPv4.

2. **Third-party tools required**: DHCPv6 on FreeBSD requires external tools such as:
   - `dhcp6` from ports (`net/dhcp6`)
   - `wide-dhcpv6` (deprecated)
   - `dhcpcd` (from ports, supports DHCPv6)
   - `kea` (from ports, primarily a DHCPv6 server but includes a client)

3. **F-Stack's user-space model**: Even if a FreeBSD DHCPv6 client were available, it would need to be ported to work within F-Stack's user-space networking model (over DPDK), where the standard socket API may not function normally.

4. **Low demand**: The primary use case for F-Stack is high-performance server applications (web servers, databases, etc.) in controlled datacenter environments where IPv6 addresses are typically statically configured.

## 5. Workarounds

### Workaround 1: Manual Configuration

Manually configure IPv6 addresses in `config.ini`:

```ini
addr6=2001:db8::100
prefix_len=64
gateway6=2001:db8::1
```

### Workaround 2: External DHCPv6 Client via KNI

Run an external DHCPv6 client (e.g., `dhcpcd` or `wide-dhcpv6`) on the KNI interface:

```bash
# Create KNI interface
./ff_proxy -c config.ini

# Run dhcpcd on the KNI interface for DHCPv6
dhcpcd -6 kni0
```

**Limitation**: This workaround is subject to the KNI limitations described in [Issue 6: MLD Multicast Reception Failure](./06-mld-multicast-reception.md), meaning DHCPv6 via KNI may not work in environments that require MLD (such as AWS EC2).

### Workaround 3: Configuration Management

Use a configuration management tool (Ansible, Chef, Puppet) to push static IPv6 addresses to F-Stack instances.
